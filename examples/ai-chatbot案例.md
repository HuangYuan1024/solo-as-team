# 实战案例：solo-as-team 在 ai-chatbot 项目的应用

> 本案例记录了 solo-as-team 工作流在 AI 智能客服系统（ai-chatbot）上的实战应用。项目为前后端分离架构：后端 Python（FastAPI + RAG + 多模态理解），前端 Vue3。

---

## 项目背景

ai-chatbot 是一个私有化部署的 AI 智能客服系统，支持文档/图片/视频上传、RAG 检索、多模态融合理解。项目由个人开发者独立完成，借助 Trae + solo-as-team 工作流在 4 天内完成多个功能迭代。

---

## 典型应用一：Lightbox 交互增强

### Step 0：需求收集

**来源**：用户反馈 lightbox 三点不足——缩放能力缺失、关闭方式局限、描述阅读受限。

**需求文档**：判定为中需求（半天工作量），写了轻量需求文档，含 3 个功能点和 11 项验收标准。

### Step 1：头脑风暴（brainstorming）

- 5 轮澄清问题确认设计细节：缩放中心=鼠标焦点、缩放范围 25%~500%、单击 vs 拖拽按 150ms 时间差区分
- 库选型对比：完全自实现 vs 引入 @panzoom/panzoom，最终选 panzoom（gzip 3.7KB，0 依赖）
- 产出 spec：328 行，含决策表 + 架构 + 数据流 + 风险 + 验收标准

### Step 2：实施计划（writing-plans）

- 719 行实施计划，8 个 Task，每个 Task 含精确文件路径和完整代码
- 小步粒度：每个 Task 拆成"写测试→验证失败→实现→验证通过→提交"

### Step 3：开发与测试

- 6 个 commit，逐步实现：Panzoom 集成、焦点缩放、指示器、pointer 时间差、自定义滚动条
- 每步 `npm run build` 验证，退出码 0 才进入下一步
- 遇到 2 个问题（见下），按降级路径处理

### Step 4：人工测试

- 端到端 11 项功能验证：滚轮缩放 / 指示器渐变 / 单击关闭 / 长按不关闭 / 描述区滚动 / Esc 关闭
- 全部通过

### Step 5：文档回补（doc-coauthoring）

- 回补 spec 实施记录章节
- 回补 plan 验证结果（11 项验证）
- 提炼 note：focal point zoom、150ms 时间窗、事件冒泡等踩坑经验
- 更新日报

## 典型应用二：DOCX 预览体验改进

### Step 0-1：需求 + 头脑风暴

- 痛点：内嵌图片展示不全、预览尺寸过小、缺少下载入口
- 设计方案：860 行单文件拆为 4 组件（Toolbar/PdfCanvas/EmbeddedGallery/容器），零后端改动

### Step 2-5：计划→开发→测试→文档

- 1945 行实施计划，8 个 Task
- 11 项端到端功能验证全过
- 产出 4 个组件 + downloadTrainingFile 工具函数
- 回补 spec + plan + note

## 工作流带来的价值

### 1. 流程纪律替代团队制衡

| 团队场景 | solo-as-team 对应 |
|----------|-------------------|
| 产品经理追问需求 | Step 0 需求文档 + 验收标准 |
| 架构师质疑方案 | Step 1 brainstorming 的 HARD-GATE + 2-3 方案对比 |
| QA 卡测试 | Step 3 分步验证 + Step 4 人工测试 |
| 技术写作写文档 | Step 5 doc-coauthoring + 读者测试 |

### 2. 文件操作铁律避免事故

开发过程中发生一起文件丢失事故（通配符 Move-Item 被 safe_rm 拦截 + Remove-Item 误删），从回收站找回后沉淀为"文件操作铁律：先迁移→再验证→后删除"，纳入 skill 的工程规则。这印证了"降级优先"和"第三方库默认行为可能隐藏致命缺陷"的规则价值。

### 3. 文档闭环避免知识流失

每个功能都有 req → spec → plan → note 完整文档链，复盘时能双向追溯：
- 从日报查到 spec → plan → 代码
- 从代码查到 plan → spec → 需求
- 从 note 查到踩坑经验

## 经验总结

1. **brainstorming 的 HARD-GATE 最有价值**：防止"想都没想就开写"，省去了大量返工
2. **writing-plans 的"禁止占位符"最实用**：计划里每个步骤都有完整代码，执行时不用再想
3. **Step 4 人工测试不可省**：Playwright 自动化测试有盲区（浏览器版本不匹配），人工测试兜底
4. **Step 5 文档回补要趁热**：开发完立刻补文档，隔几天就忘了细节
5. **工作流是循环的**：复盘产生的新需求（项目改进与提升扩展计划）会回到 Step 0 开启下一轮
