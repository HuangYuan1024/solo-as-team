# 个人即团队 Skill（Solo as Team）

一个人 + AI = 一个完整的开发团队。本 skill 让个人开发者借助 AI 完成原本需要产品、设计、开发、测试、文档等多个角色协作的全流程工作，覆盖需求→设计→实施→测试→文档的全生命周期。融合了 brainstorming、writing-plans、doc-coauthoring、test-driven-development、executing-plans、git-commit 六个 skill 的调用时机，以及一套通用工程规则。

**核心理念**：AI 不是替代某个角色，而是让一个人能同时扮演所有角色。流程纪律 + 工程规则是保证质量的关键——团队靠协作制衡，个人靠流程制衡。

## 一、工作流总览（6 步闭环）

```
0. 需求 -> 1. 头脑风暴 -> 2. 实施计划 -> 2.5 计划审查 -> 3. 开发与测试 -> 4. 人工测试 -> 5. 文档回补 -> 复盘产生新需求，回到步骤0
```

**核心原则**：
- 流程是循环的，复盘产生的新需求回到 Step 0 开启下一轮
- 第 0 步是入口，需求文档是头脑风暴的明确输入，避免凭空设计
- 第 2.5 步是计划审查闸门，未通过审查的计划不进入开发
- 第 4 步是人工验收，承认 AI 在 UX/视觉/端到端验收上的盲区
- 第 5 步是"回补"而非"先写"，文档跟随现实，避免失真

### 1.1 流程级 TDD（测试驱动开发）

TDD 的本质不是"写测试"，而是**"先定义会失败的验收标准，再做实现，最后验证标准被满足"**。这套思路从代码层提升到流程层，每个阶段都遵循 Red-Green-Refactor：

| 阶段 | Red（先定义会失败的验收标准） | Green（做实现让标准通过） | Refactor（优化，保持验收通过） |
|------|------------------------------|--------------------------|-------------------------------|
| Step 0 需求 | 验收 checklist（功能没做，验收必失败） | 写需求文档 | 补非功能需求、关联文档 |
| Step 1 设计 | 设计审查清单 + 降级矩阵（设计没写，审查必失败） | 写 spec | 按审查清单修正 |
| Step 2 计划 | 计划审查清单（计划没写，审查必失败） | 写 plan | 按审查清单修正 |
| Step 2.5 审查 | 严重度分级问题清单（P0 必为 0） | 逐项修正 | 第二轮查上一轮副作用 |
| Step 3 代码 | 单元测试（传统 TDD） | 最小实现 | 重构 + 代码审查 |
| Step 4 人工测试 | 复用 Step 0 验收 checklist | 逐条验收 | 修 bug |
| Step 5 文档 | 读者测试问题（doc-coauthoring SKILL 已有） | 写文档 | 按读者反馈改 |

**铁律：必须"先看它失败"**。每阶段的验收标准先空着跑一遍，确认每条都"不通过"，再做实现。这个动作是防止自检流于形式的关键--写完顺手扫一眼不叫自检，拿着清单逐条查才算。

**验收链闭环**：Step 0 定义的验收 checklist 必须在 Step 4 被逐条复用，验收标准不能在流程中途漂移。

### 1.2 是否启动循环的三问筛子（前置闸门）

本 skill 的 6 步闭环本身就是一个"人驱动的循环"。但**不是所有需求都值得套重流程**。启动工作流前，先用三问筛子判断这个任务是否值得做成"循环"来工程化（来自 Loop Engineering 的黄金规则）：

| # | 问题 | 全否时的处理 |
|---|------|-------------|
| 1 | 这个任务**每周至少重复一次**？ | 否=一次性任务，不值得建循环 |
| 2 | 有**可自动验证**的完成标准？ | 否=纯主观判断，不适合循环 |
| 3 | **每次执行的结果会改变下一次怎么做**？ | 否=只是定时任务（cron job），不需要循环工程化 |

**判定规则**：
- 三问全"是" -> 值得做成自适应循环，完整走 6 步并把终止/回退/状态护栏（见 3d、4.4）落实
- 部分否 -> 按下方需求分层阈值降级处理
- 三问全否 -> 一次性任务，直接做，日报记一笔即可

> 核心判据：**只有"一轮的结果改变下一轮的行动"才是循环**。第 N 次跑完的结果不影响第 N+1 次怎么跑，那就是 cron job，不需要 Loop Engineering，也别套 6 步重流程。

三问筛子与需求分层阈值（见 Step 0）配合使用：三问筛子判断"是否值得循环化"，分层阈值判断"流程该多重"。

### 1.3 Loop Engineering 视角下的定位

本 skill 实现的是 **Harness Engineering** 层--单轮运行的运行时基础设施（工具/权限/上下文）+ 流程纪律 + 工程规则。2026 年 6 月兴起的 **Loop Engineering（循环工程）** 是建在 Harness 之上的顶层编排层，聚焦"Context（每轮给 AI 看什么）+ Lifecycle（循环启停）"两层。二者是层叠关系，不是替代关系：**没有 Harness 的 Loop 就像没有刹车的跑车**。

本 skill 从 v2 起将 Loop Engineering 的思维与护栏**融入设计时、沉淀为运行时约束**（STATE.md 跨循环状态、终止/回退护栏、验证阶梯、Maker-Checker、Improvement Loop），而非新建一个 loop 文档塞进上下文。落地时牢记三条认知纠正：

1. **设计时 vs 运行时**：Loop 框架的价值在于"迫使系统思考"（12 问、终止条件、验证阶梯），运行时执行逻辑应写在该在的地方--skill（步骤）/ hooks（护栏）/ 脚本（自动化）。**不要新建 loop spec 文件**。
2. **让循环笨而可靠，让基础设施聪明**：循环决策逻辑应占比极小，复杂度放在支撑循环的基础设施（验证、状态、回退）上。循环骨架保持简单，警惕把流程越搞越"clever"。
3. **Loop ≠ cron job，也 ≠ 现有的"复盘回环"**：只有"一轮结果改变下轮行动"才是 Loop。现有"复盘->Step0"是人驱动的宏观回路；Step 3 内的 Red-Green-Refactor 是自适应微观循环的雏形。分清二者，别把所有"重复"都误当 Loop 来工程化。

## 二、各步骤详解

### Step 0：需求收集与确认

**输入**：用户反馈 / 自己发现 / 复盘改进点
**产出**：需求文档（req/ 目录）

**需求分层阈值**（避免过度设计）：
| 需求规模 | 处理方式 |
|----------|----------|
| 小需求（<1h，改按钮/调样式） | 不写文档，直接做，日报记一笔 |
| 中需求（半天~2天） | 写轻量需求文档 |
| 大需求（新功能/重构） | 写完整需求文档 + 评审 |

**需求文档必备内容**：
- 需求背景（要解决什么问题、谁受益、不做的代价）
- 功能点拆解
- 验收标准（可测试的 checklist）
- 非功能需求（性能/兼容/降级，没有可删）
- 关联文档（实施后补 spec/plan 链接）
- 需求变更记录（保留变更痕迹）

### Step 1：头脑风暴

**调用 skill**：`brainstorming`
**输入**：需求文档 + 现有代码
**产出**：设计方案（spec/ 目录）

**brainstorming 的硬性约束**：
- HARD-GATE：设计批准前不写任何代码，不论项目多简单
- 一次只问一个问题，多选优先
- 必须提 2-3 个方案对比，带 trade-offs 和推荐
- 设计分段呈现，每段确认后再继续
- spec 自检：占位符扫描、内部一致性、范围检查、歧义检查
- 用户审查 spec 后才转入 writing-plans

**spec 必备产出（流程级 Red）**：
- **降级与容错矩阵**：每个模块明确标注失败场景、降级策略、fail 模式（fail-open/fail-closed/fail-skip）。比笼统说"降级优先"可执行得多，是 Step 3 开发降级路径的验收标准。

  | 模块 | 失败场景 | 降级策略 | fail 模式 |
  |------|---------|---------|----------|
  | 例：限流器 | 内存异常 | 记日志放行 | fail-open |
  | 例：鉴权 | DB 不可用 | 返回 503 | fail-closed |

- **设计审查清单**：列出本设计必须满足的硬性约束（如"所有远程调用必设超时"、"CORS 中间件最先注册"），作为 Step 2.5 计划审查的输入。
- **三类信息标注（防认知投降）**：spec 与后续文档中所有结论性陈述必须标注类型，防止把推断写成事实、误把 AI 自信当事实：
  - **事实**：代码/日志/文档直接可见的信息
  - **推断**：基于证据做出的判断（须附证据）
  - **不确定**：证据不足、需人工确认的地方（须标明缺什么证据）
  此标注在 Step 4 人工测试时优先核查"推断"与"不确定"项，是对抗认知投降的实务手段。

**探索项目上下文**：
- 动手前核实代码现状，避免盲目按计划开发重复造轮子
- 检查现有文件、文档、最近提交
- 在现有代码库中，先探索结构再提改动，遵循现有模式

### Step 2：实施计划撰写

**调用 skill**：`writing-plans`
**输入**：设计方案 + 现有代码
**产出**：实施计划（plan/ 目录）

**writing-plans 的硬性约束**：
- 假设执行者对代码库零上下文
- 小步任务粒度（2-5分钟一步）：写失败测试->验证失败->最小实现->验证通过->提交
- 禁止占位符：不许"TBD/TODO/类似Task N/添加适当错误处理"
- 每步含精确文件路径、完整代码、精确命令和预期输出
- 自检：spec 覆盖率、占位符扫描、类型一致性
- DRY / YAGNI / TDD / 频繁提交

### Step 2.5：计划审查（质量闸门）

**输入**：实施计划 + 设计审查清单 + 降级矩阵
**产出**：审查修正记录（回填到 plan/ 和 spec/）

计划写完不直接进开发，必须先过审查闸门。审查是 Step 2 的 Red--计划没写时审查必失败，写完后拿着清单逐条查。

**严重度分级标准**：

| 级别 | 含义 | 处理 |
|------|------|------|
| P0 | 阻断性问题（安全漏洞、数据丢失、并发竞态、性能灾难） | 必须立即修正，不修正不进开发 |
| P1 | 影响质量（伪 Task、缺依赖、测试不可执行、降级缺失） | 必须修正 |
| P2 | 优化项（命名、顺序、循环导入风险） | 建议修正 |

**必须做两轮审查**：
- **第一轮**：按下方维度清单逐条查，记录所有问题及严重度
- **第二轮**：专门查第一轮修正引入的副作用（如改了方法名后测试 fixture 还在用旧名、删了变量后 conftest 还在引用）

**Maker-Checker（制造者/检查者分离）**：写代码/计划的 Agent 不能评判自己的工作--就像作家不能同时当自己的编辑。Step 2.5 与 Step 3a 的审查应派**独立子 Agent**（不带前序设计与开发上下文）来做，只给它计划/代码 + 审查清单，让它返回问题清单。这是无人值守循环的基石，也是"个人靠流程制衡"的多一道独立制衡。若环境不支持子 Agent，则由人在审查环节充当 Checker，但必须在文档中标注"自审而非独立审"，质量保障降级。

**审查维度清单**（逐条对照，不可跳过）：
1. 安全性：鉴权越权、scope 校验、敏感信息泄露
2. 并发竞态：状态机中间态、共享变量、asyncio.Lock 覆盖
3. 性能：N+1 查询、逐库检索、高频小写入
4. 契约一致性：对外接口、错误响应格式、request_id 传递
5. 测试可执行性：依赖是否齐全、fixture 是否引用未实现变量、mock 策略
6. 降级路径：每个模块的 fail 模式是否明确、是否与降级矩阵一致
7. 配置驱动：可变参数是否走配置、有无硬编码
8. 循环导入：跨模块引用是否安全
9. 中间件顺序：CORS、Auth、RequestContext 的注册顺序
10. 占位符扫描：有无 TBD/TODO/伪 Task
11. **预算与成本上限**：任务/循环是否设了最大轮数、最大预算、超时、失败次数上限。未设预算的循环是"烧钱的死循环"，P0 阻断
12. **终止与回滚策略**：终止条件是否明确（目标达成/资源耗尽/质量劣化/主动中断四类，见 3d）、回退策略是否选定（Ratchet/Rollback/Retry/Branch）。终止条件说不清的，不进开发
13. **日志审计**：关键动作（提交、部署、数据写入、权限提升）是否可审计回溯；STATE.md 是否记录了"上次做到哪/上次结果"。无审计=无法复盘=无法改进

> verifier 质量自检（反馈信号质量）：审查时除查"有没有验证"，还要查"验证本身对不对"--测试覆盖是否完整、会不会让 Agent 只修到测试通过（优化错误目标）、LLM 判断会不会阻止正确改动。**循环的核心不是循环次数，而是反馈信号质量**。

**审查记录回填**：审查发现的问题、修正方案、落地状态（计划已修正/Task X 落地）全部回填到 spec 的实施记录章节，作为 Step 5 文档回补的输入。

### Step 3：开发与测试（Trae 主导）

**调用 skill**：`test-driven-development` + `executing-plans` + `git-commit`

**3.0 TDD 铁律（集成 test-driven-development SKILL）**：

- **铁律**：NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST（没有失败测试就不写生产代码）。先写代码后补测试不算 TDD，必须删掉重写。
- **watch it fail 强制**：写完测试必须先运行确认它失败，且失败原因正确（功能缺失，不是拼写错误）。测试立即通过说明测的是已有行为，没价值。
- **Red-Green-Refactor 循环**：写一个失败测试 -> 看它失败 -> 写最小实现让它通过 -> 看它通过 -> 重构（保持绿）-> 下一个。
- **反模式禁令**（来自 testing-anti-patterns）：不测 mock 行为而测真实行为、不给生产类加 test-only 方法、不盲目 mock 不理解的依赖。
- **bug 修复必须先写复现测试**：发现 bug 先写一个能复现它的失败测试，再修，测试通过即修复完成且防回归。

**3.1 stop-and-ask 协议（集成 executing-plans SKILL）**：

遇到以下 blocker **立即停下问人，不猜、不硬闯**：
- 测试反复失败（修了 2 次还不通过）
- 依赖缺失（缺包、缺服务、缺配置）
- 指令歧义（计划步骤有多种理解）
- 验证失败超 2 次（说明理解有偏差）
- 计划有关键缺口（缺步骤、缺文件路径）

停下时回到 Step 2.5 审查计划缺口，或直接向用户澄清。撞墙继续猜是最大的时间浪费--今天开发中遇到的 PowerShell `&&` 语法、虚拟环境无 pip、局部导入无法 patch，本质都是撞墙时该停下来确认环境而非继续猜。

**3.2 开发原则**：
- 每开发一步就验证测试一步，禁止多步堆积后统一测试
- 坚持代码复用（新增代码前排查现有能力是否可复用）
- 保证代码最小改动（不顺手重构无关代码，保持 diff 可审）
- 设计保守的降级兜底策略（增强功能失败不阻断主流程）
- 契约不变（重构时保持对外接口不变，只改内部实现）

**3a 代码审查（每个部分开发完后）**：

每完成一个部分（如 Task 0-8 一组），立即做一轮轻量代码审查，抓单元测试抓不到的问题。审查维度：

| 维度 | 检查点 |
|------|--------|
| 错误响应格式 | 是否统一结构化（code/msg/data），有无半成品 |
| 上下文传递 | request_id、caller 等是否在所有错误路径都传递，有无传空串 |
| 测试 fixture | 是否引用了未实现的变量、是否污染全局状态 |
| 导入方式 | 被测试 patch 的依赖是否用模块级导入（非方法内局部导入） |
| 降级一致性 | 代码实现是否与 Step 1 降级矩阵的 fail 模式一致 |

发现问题立即修正，不堆积到 Step 5。

> 本审查遵循 Maker-Checker（见 Step 2.5）：尽量由独立子 Agent 执行，不带开发上下文，只拿代码 + 审查维度表返回问题清单。

**3b 遗留待办追踪**：

不能立即修复的问题不丢弃，记成待办表，每条标注：

| 字段 | 说明 |
|------|------|
| 问题描述 | 现象 + 根因 |
| 影响范围 | 哪些功能/测试受影响 |
| 计划落地位置 | 具体到 Task 号（如"Task 14 实现 check_rate_limit 时补上"） |

待办表在 Step 5 文档回补时同步到 spec 的实施记录"遗留待办"章节。这是保证问题不丢、准确传递到后续 Task 的关键机制。

**3c 提交规范与 Git 安全协议（集成 git-commit SKILL）**：

每个 Task 完成且测试通过后立即提交，遵循 Conventional Commits：

```
<type>[scope]: <description>

[optional body]
```

| 约束 | 要求 |
|------|------|
| type | feat/fix/docs/style/refactor/perf/test/build/ci/chore/revert |
| description | 现在时祈使句（add 非 added），<72 字符，`可询问使用何种语言编写（默认中文）` |
| 暂存粒度 | 一个逻辑改动一个 commit，不混无关改动 |
| 引用 | 关联 issue 用 `Closes #123` / `Refs #456` |

**Git 安全协议（硬约束）**：
- 永不修改 git config
- 永不执行破坏性命令（--force、hard reset）除非用户明确要求
- 永不跳过 hooks（--no-verify）除非用户要求
- 永不 force push 到 main/master
- hook 失败时修复后建新 commit，不 amend
- 永不提交密钥（.env、credentials.json、私钥）

**3d 终止条件与回退策略（循环护栏）**：

开发是一个反复迭代的过程（Red-Green-Refactor 本质就是自适应微观循环）。任何循环都必须先定"何时停、出错怎么办"，否则就是烧钱/烧时间的死循环。

**开始循环前三个自问**：
1. 终止条件是什么？答不上来，不要开始
2. 最坏情况下会发生什么？不可接受，加检查点（人工闸门）
3. 产出结果我能验证吗？不能，这个任务不适合自动化循环

**四类终止条件**（至少命中一类才停）：

| 终止类型 | 含义 | 本 skill 实例 |
|----------|------|--------------|
| 目标达成 | 验收 checklist 全绿 / 测试全过 | Step 0 验收 checklist 全绿、Step 3 单元测试全过 |
| 资源耗尽 | 超出预算上限 | 达到最大轮数 / 最大 token / 超时 / 失败次数上限 |
| 质量劣化 | 产出质量可量化且在下降 | 连续 N 次修复无改进（说明方向错了） |
| 主动中断 | 人类介入 | stop-and-ask 触发、用户喊停 |

**五个命名终止状态**（决定后续处理）：`success`（完成）/ `no-op`（无事可做）/ `blocked`（被阻断，需人介入）/ `stalled`（卡住无进展）/ `exhausted`（预算耗尽）。每个状态对应不同后续：success->进下一步；blocked/stalled->回 stop-and-ask；exhausted->降级交付或重设预算。

**四类回退策略**（决定容错模式）：

| 策略 | 机制 | 适用场景 |
|------|------|----------|
| **Ratchet（只前进）** | 改进才 commit、没改进就 revert | 目标函数明确且单调（如测试通过率） |
| **Rollback（回滚）** | 恢复到上一个良好状态 | 操作有副作用且不可重复（如数据写入） |
| **Retry（重试）** | 原地重新执行 | 失败是随机性的（如网络抖动） |
| **Branch（分支）** | 多方案并行尝试 | 解空间大且资源充足（git worktree 并行实验） |

> Ratchet 与流程级 TDD 同构：每轮迭代必须通过验证阶梯（测试/lint/类型检查）才落盘，没过就 revert--把 Red-Green 升级为循环层的"改进才前进"。若目标不是清晰标量（如"代码质量"），Ratchet 不适用，改用 Rollback + 人工审查。

**预算护栏（硬约束）**：所有循环必须设最大轮数、最大预算（token/时间）、超时、失败次数上限。预算耗尽时按 exhausted 状态降级，绝不无限重试。

### Step 4：人工测试（人主导）

**范围**：UX/视觉/真实浏览器/复杂样本回归

**验收链闭环（强制）**：人工测试必须**逐条复用 Step 0 的验收 checklist**，不能临时想验收标准。Step 0 定义的每条验收标准在此阶段逐条跑，全绿才算通过。这是"流程级 TDD"的最后一环--需求阶段定义的失败标准，到人工测试阶段必须全部变绿。

**验证阶梯（Verification Ladder）**：本 skill 的验证是分层的，人工测试是阶梯顶层，不可跳过底层只靠人工兜底：

| 层级 | 验证手段 | 自动/人工 | 对应步骤 |
|------|----------|----------|----------|
| L1 | git 退出码 / 构建退出码 | 自动门控 | Step 3 |
| L2 | lint / typecheck | 自动 | Step 3 |
| L3 | 单元测试 / 集成测试 | 自动 | Step 3（TDD） |
| L4 | 独立子 Agent 抽查（Maker-Checker） | 半自动 | Step 2.5 / 3a |
| L5 | 人工验收（复用 Step 0 checklist） | 人工 | Step 4 |

> 阶梯原则：底层自动门控不通过的，不进入上层。L1-L3 全绿才提交 L4 审查，L4 通过才进 L5 人工。警惕"审批肌肉记忆"--权限/合并弹窗批准率过高时，底层必须做沙箱隔离和严格权限分级，不能把安全性完全押宝在"用户会仔细审查"。

**反馈回路**：
- 发现 bug -> 回 Step 3 修复
- 发现设计缺陷 -> 回 Step 1 重新设计
- 发现需求偏差 -> 回 Step 0 补需求

### Step 5：文档撰写与回补（Trae 主导）

**调用 skill**：`doc-coauthoring`
**产出清单**：
1. 回补 spec/ 实施记录章节（与原设计差异、验证结果、踩坑经验）
2. 回补 plan/ 验证结果（任务完成状态、偏离记录、验证项）
3. 从日报"备忘/学习"提炼到 note/
4. 更新日报
5. 复盘产生新需求 -> 记入 req/ 或项目改进路线图
6. **Improvement Loop（改进信号回填）**：把本轮历史结果变成下一轮的改进信号，回填到 STATE.md 的"改进信号"区与本 skill 自身：
   - 哪些 bug 反复出现 -> 下一轮 Step 2.5 审查清单前置检查
   - 哪些测试常假绿（优化错误目标）-> 加强 verifier / 补测试覆盖
   - 哪些指令常歧义 -> 补 SKILL.md 或基础 skill 措辞
   - 哪些降级路径被真实触发 -> 回补降级矩阵实战经验
   这是 Loop Engineering 最高层（Improvement Loop）：让系统越跑越准，而非每次从零推导。**只有"一轮结果改变下轮行动"才是循环**，改进信号回填就是把这个原则落到文档。

**doc-coauthoring 的三阶段**：
- 上下文收集：提供所有相关上下文，澄清问题
- 精炼与结构：逐节构建，澄清→头脑风暴→策展→缺口检查→起草→迭代
- 读者测试：预测读者问题，用无上下文的 agent 测试文档可读性

## 三、通用工程规则

> 以下规则从项目实战中提炼，适用于所有开发任务。

### 3.1 工程方法

- **流程级 TDD（测试驱动开发）**：TDD 的本质是"先定义会失败的验收标准，再做实现，最后验证标准被满足"。这套思路从代码层提升到流程层--需求阶段写验收 checklist，设计阶段写降级矩阵和审查清单，计划阶段做两轮审查，代码阶段写单元测试，人工测试阶段复用需求 checklist。每阶段的"绿"都对应前一阶段的"红"被消除，形成贯穿全流程的验收链。详见第一章 1.1 节。
- **分步验证**：功能拆为最小步骤，每步立即测试验证，禁止多步堆积后统一测试
- **两轮审查**：计划审查必须做两轮，第一轮按维度清单查问题，第二轮专门查第一轮修正引入的副作用
- **降级优先**：先设计失败场景和降级路径，增强功能失败不阻断主流程
- **复用优先**：新增代码前排查现有能力是否可复用，优先复用而非重写
- **最小改动**：只做必要改动，不顺手重构无关代码，保持 diff 可审
- **契约不变**：重构时保持对外接口不变，只改内部实现，上层无感知
- **配置驱动**：可变参数走配置（并发度、超时、阈值、模型名、服务地址），不硬编码
- **遗留待办不丢**：不能立即修复的问题记成待办表（问题+影响+计划落地位置），传递到后续 Task 和 Step 5 文档回补
- **动手前核实代码现状**：避免盲目按计划开发重复造轮子

### 3.2 代码规范

- 所有回答使用中文表述，代码注释用中文
- 函数/方法代码超过 30 行时，优先考虑是否可以抽象或聚合
- 避免不必要的对象复制或克隆
- 避免多层嵌套，提前返回
- 使用适当的并发控制机制
- 命名使用有意义的描述性名称，避免缩写和单字母变量（循环中的 i 除外）
- 相关代码放在一起，函数只做一件事，保持适当的抽象层次
- **测试可 patch 性**：被测试 patch 的依赖必须用模块级导入，禁止方法内局部导入（`from x import y` 写在函数体内）。局部导入会让 patch 命中模块级引用而测试无效，除非确有循环导入风险并显式说明

### 3.3 健壮性与容错

- **降级路径设计**：增强功能失败逐级降级，最终总能有可用输出
- **超时必设**：所有远程/本地调用都设超时，避免无限等待
- **错误隔离**：单点失败不影响全局，批次失败不影响其他批次
- **异常分层处理**：可恢复异常降级继续，不可恢复异常抛上层，预期失败返回空串继续
- **幂等设计**：重复操作无副作用
- **去重处理**：时间戳去重、文本相似度去重、内容完全相同去重
- **第三方库默认行为可能隐藏致命缺陷**：生产使用前必须压测验证，不信任默认配置
- **并发状态机防竞态**：状态机的中间态（如熔断器 half_open）必须用独立标志位 + 锁限制并发，不能只靠状态字段判断。否则多个请求同时进入中间态会绕过保护（如多个探测请求同时通过 half_open）。C 扩展模型（PaddleOCR/FunASR 等）非线程安全，多线程并发会崩溃，必须串行化推理

### 3.4 性能优化

- 避免不必要的对象创建，及时释放不再需要的资源，注意内存泄漏
- 避免重复计算，使用适当的数据结构和算法，延迟计算直到必要时
- 识别可并行化的任务，避免不必要的同步，注意线程安全问题
- **高频小写入用攒批异步**：每次调用都写 DB 的场景（如更新 last_used_at、计数器）用"攒批 + 异步刷盘"模式（如 30s 攒批后台写入），并设队列上限防止内存暴涨。降低 DB 压力又不丢数据
- **配置热加载范式**：可热加载的配置（开关、白名单、限流阈值）走 `runtime_config`（frozen dataclass + yaml，文件修改后下次请求生效，无需重启）；不可热加载的（密钥、DB 连接）走 `settings` 启动加载。两类配置分层管理，避免混淆
- **单一数据源原则**：同一计数器/状态不要内存和 DB 双写（会冲突），统一到一个数据源（如配额计数统一走 DB，内存只读缓存）

### 3.5 文件操作铁律

**涉及文件移动/重命名/删除等不可逆操作时，必须分三步执行**：

| 步骤 | 动作 | 验证标准 | 禁止事项 |
|------|------|----------|----------|
| 1. 迁移 | 把文件从 A 复制/移动到 B | — | 不删 A |
| 2. 验证 | 列出 B 的文件，核对数量、字节数、可读性 | 数量一致 + 字节数一致 + 能打开 | 不假设成功 |
| 3. 确认 | 人工确认无误（或自动断言通过） | — | 不自作主张进步骤 4 |
| 4. 清理 | 确认后才删 A | — | 不用 -Recurse -Force 暴力删未验证目录 |

大改动加备份：批量文件操作前，先复制整个目录到 .tmp/ 或 .bak/，双保险。

## 四、文档规范

### 4.1 文档分层

| 层级 | 目录 | 职责 |
|------|------|------|
| **运行态状态** | `{项目}/STATE.md` | **当前在做什么/上次结果/等人工**（跨步骤跨循环，每步先读后写，详见 4.4） |
| 需求文档 | `{项目}/docs/req/` | 做什么/为什么 |
| 设计文档 | `{项目}/docs/spec/` | 怎么做（方案设计、架构决策） |
| 实施计划 | `{项目}/docs/plan/` | 何时做（任务分解、步骤） |
| 技术笔记 | `{项目}/docs/note/` | 怎么做好（踩坑、复盘、学习） |
| 项目级路线图 | `{项目}/docs/项目改进与提升扩展计划.md` | 跨多功能改进计划 |
| 日报（可选） | `{项目}/docs/daily/YYYY-MM/YYYY-MM-DD.md` | 做了什么（每日进度） |

> STATE.md 是"运行态"，其余是"归档态"。归档态是产物沉淀，运行态回答"现在到哪了"。两者不可混用：不要把 STATE.md 当设计文档写长篇，它只记三个问题的答案。

### 4.2 文档质量要求

- 设计文档必备字段：创建日期、状态、适用项目、关联文档
- 设计文档实施后回补记录：与原设计差异、验证结果、踩坑经验
- 设计文档实施记录必备子章节：
  - **总体进度表**：各阶段范围、状态、测试数、备注
  - **审查修正记录**：第一轮 + 第二轮问题及落地状态
  - **P0/P1 问题落地追踪表**：每个问题的实现位置 + 验证方式（哪个测试文件）
  - **Bug 记录**：按"现象-根因-处理"三段式记录
  - **遗留待办**：问题 + 影响范围 + 计划落地位置（Task 号）
- 学习笔记主题单一：按问题域划分，笔记间可交叉引用但避免内容重复
- 日报问题记录完整：现象-根因-处理，供后续复盘
- 代码改动后同步修正过时文档，避免文档与代码不一致
- 文档间交叉引用形成知识图谱，使用相对路径链接
- 文档提交前自检：让陌生人阅读验证是否可理解

**Bug 记录模板**（日报和 spec 实施记录通用）：

```markdown
### X.X Bug 标题
- **现象**：观察到的问题表现
- **根因**：导致问题的代码/设计原因
- **处理**：具体的修正方案
```

### 4.3 注释规范

- 注释解释"为什么"而非"做什么"
- 函数头必备功能、参数列表、返回值说明
- 严禁行内注释，函数体内必须使用行间注释
- 为公共 API 提供清晰的文档
- 及时更新注释以反映代码变化

### 4.4 STATE.md：跨步骤跨循环的运行态状态文件

模型没有跨会话长期记忆，长对话会丢上下文，但状态文件不会。STATE.md 是本 skill 落地 Loop Engineering "Memory 原语"的载体，回答三个关键问题：**当前在做什么？上次结果如何？什么在等人工处理？**

**使用规则**：
- 每个步骤开始时**先读** STATE.md，恢复上下文（避免凭记忆续接）
- 每个步骤结束时**更新** STATE.md，记录进展与待办
- 跨日/跨会话恢复工作时，第一件事是读 STATE.md
- STATE.md 只记运行态，不替代 spec/plan 等归档文档

**STATE.md 最小结构**：

```markdown
# 项目运行状态

## 当前阶段
Step X：xxx（一句话）

## 上次结果
- 完成了：...
- 验证状态：success / blocked / stalled / exhausted
- 遗留问题：...

## 等人工处理
- [ ] ...

## 改进信号（供 Improvement Loop，见 Step 5）
- 哪些 bug 反复出现 -> 下次前置检查
- 哪些测试常假绿 -> 加强 verifier
- 哪些指令常歧义 -> 补 skill
```

> "Agent 会遗忘，仓库不会。" STATE.md 让循环跨会话延续，也让人能随时介入接管。它与遗留待办表（3b）的区别：待办表记"代码层未解决问题"，STATE.md 记"流程层整体进度"。

**简化降级流程**（用户确认知情且坚持在缺失 skill 下开发时）：
- 缺 `brainstorming`：Step 1 改为直接对话式澄清需求，但无 HARD-GATE 约束
- 缺 `writing-plans`：Step 2 改为输出简化任务清单，无小步粒度和占位符检查
- 缺 `test-driven-development`：Step 3 仍要求先写测试，但无 watch it fail 强制和反模式禁令
- 缺 `executing-plans`：Step 3 仍要求撞墙停下，但无明确的 blocker 判定清单
- 缺 `git-commit`：Step 3 仍要求频繁提交，但无 Conventional Commits 规范和 Git 安全协议
- 缺 `doc-coauthoring`：Step 5 改为直接按模板填写文档，无读者测试环节

## 六、使用方式

当用户提出新功能开发、项目任务、或提到"开发/实现/做一个功能"时，按以下方式启动工作流：

1. **前置依赖检查**：按第五章检查六个基础 skill 是否已安装，缺失则提示下载，不继续
2. **三问筛子**（见 1.2）：判断这个任务是否值得做成"循环"来工程化。三问全否=一次性任务，直接做不套重流程
3. **读 STATE.md**（见 4.4）：恢复上次进度，避免凭记忆续接；若不存在则创建
4. 判断需求规模（小/中/大），决定是否需要写需求文档
5. 如果需要，先在 `dev-docs/{项目}/req/` 下创建需求文档
6. 调用 `brainstorming` skill 进入 Step 1
7. brainstorming 完成后自动转入 `writing-plans` skill 进入 Step 2
8. Step 2.5 计划审查（Maker-Checker 独立审查，13 条维度）通过后，按 plan 逐步开发（Step 3，遵循 TDD 铁律 + stop-and-ask + 3d 终止/回退护栏 + git-commit 规范）
9. 提醒用户进行人工测试（Step 4，验证阶梯 L5）
10. 开发完成后调用 `doc-coauthoring` skill 回补文档（Step 5，含 Improvement Loop 改进信号回填）
11. **更新 STATE.md**：记录本轮结果与遗留待办，供下一轮循环续接

**工作流不是线性的，是循环的**：任何步骤发现问题都可以回到前序步骤，复盘产生的新需求回到 Step 0。每一轮的改进信号（Step 5 第 6 项）回填后，下一轮的审查清单与 skill 会越跑越准。
